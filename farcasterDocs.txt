# Farcaster Mini Apps

## Why Farcaster Doesn't Need OAuth 2.0

OAuth exists to let three separate parties (user → platform → third-party app) establish mutual trust. Farcaster is built on a decentralized architecture that collapses this triangle:

### 1. Identity & Authentication
- **User-owned keys:** A user controlled cryptographic signature proves control of a Farcaster ID—no intermediary.
- Sign In with X → Sign-in with Farcaster (SIWF)
- OAuth 2.0 Authorization Flow → Quick Auth

### 2. Data Access & Permissions
- **Open, replicated data:** Social data like casts, reactions, and profiles live on Snapchain and can be read by anyone.
- **No permission scopes:** Everything is already public; you filter what you need instead of requesting scopes.
- **Zero-cost reads:** Sync the chain yourself or hit a public indexer—no rate caps, no $5k+/month fire-hoses.
- **Cryptographic writes:** Users can delegate a key to applications so the applications can write on their behalf.
- Centralized APIs → Snapchain + infra services (e.g. Neynar)
- Access token → no equivalent, data is public
- Write permissions → App Keys

### Builder Takeaways
1. Skip OAuth flows—wallet signature = auth.
2. Forget permission scopes—use filters.
3. Enjoy building permissionlessly

## Getting Started

### Overview
Mini apps are web apps built with HTML, CSS, and Javascript that can be discovered and used within Farcaster clients. You can use an SDK to access native Farcaster features, like authentication, sending notifications, and interacting with the user's wallet.

### Requirements
- **Node.js 22.11.0 or higher** (LTS version recommended)
- A package manager (npm, pnpm, or yarn)

> ⚠️ If you encounter installation errors, verify you're using Node.js 22.11.0 or higher. Earlier versions are not supported.

### Enable Developer Mode
1. Make sure you're logged in to Farcaster on either mobile or desktop
2. Go to https://farcaster.xyz/~/settings/developer-tools
3. Toggle on "Developer Mode"
4. A developer section will appear on the left side of your desktop display

### Quick Start (CLI)
```bash
# npm
npm create @farcaster/mini-app

# pnpm
pnpm create @farcaster/mini-app

# yarn
yarn create @farcaster/mini-app
```

### Manual Setup

#### Package Manager
```bash
npm install @farcaster/miniapp-sdk
pnpm add @farcaster/miniapp-sdk
yarn add @farcaster/miniapp-sdk
```

#### CDN
```html
<script type="module">
  import { sdk } from 'https://esm.sh/@farcaster/miniapp-sdk'
</script>
```

### Making Your App Display (Ready Signal)
**CRITICAL**: After your app loads, you MUST call `sdk.actions.ready()` to hide the splash screen and display your content. If you don't call `ready()`, users will see an infinite loading screen.

```javascript
import { sdk } from '@farcaster/miniapp-sdk'

// After your app is fully loaded and ready to display
await sdk.actions.ready()
```

## Specification

### Manifest File Format

The manifest is hosted at `/.well-known/farcaster.json` on your app's domain.

#### Full Manifest Schema (TypeScript)
```ts
type Manifest = {
  accountAssociation: {
    header: string;      // base64url encoded JFS header
    payload: string;     // base64url encoded payload with domain
    signature: string;   // base64url encoded signature bytes
  };
  miniapp: MiniAppManifest;
};

type MiniAppManifest = {
  version: "1";
  name: string;                    // max 32 chars
  iconUrl: string;                 // max 1024 chars, min 200x200px, max 1024x1024px
  homeUrl: string;                 // max 1024 chars
  imageUrl?: string;               // max 1024 chars, 3:2 ratio recommended
  buttonTitle?: string;            // max 32 chars
  splashImageUrl?: string;         // max 1024 chars, 200x200px
  splashBackgroundColor?: string;  // hex color e.g. #eeccff
  webhookUrl?: string;             // max 1024 chars
  description?: string;            // max 170 chars
  primaryCategory?: AppCategory;
  tags?: string[];                 // max 5 tags, max 20 chars each
  heroImageUrl?: string;           // max 1024 chars, 1.91:1 ratio recommended
  tagline?: string;                // max 30 chars
  ogTitle?: string;                // max 60 chars
  ogDescription?: string;          // max 160 chars
  ogImageUrl?: string;             // max 1024 chars
  noindex?: boolean;               // exclude from search/discovery
  requiredChains?: string[];       // CAIP-2 chain identifiers
  requiredCapabilities?: string[]; // SDK method paths
  canonicalDomain?: string;        // for domain migration
};
```

```ts

  | "education"
  | "developer-tools"
  | "entertainment"
  | "art-creativity";
```

### Embed Format (fc:miniapp Meta Tags)

Embeds allow individual pages to be shared as rich cards in social feeds.

```html
<meta property="fc:miniapp" content="..." />
```

The content is a JSON string with the following schema:

```ts
type MiniAppEmbed = {
  version: "next";
  imageUrl: string;        // 3:2 ratio recommended, max 1024 chars
  button: {
    title: string;         // max 32 chars
    action: {
      type: "launch_miniapp";
      name: string;        // max 32 chars
      url: string;         // max 1024 chars
      splashImageUrl?: string;
      splashBackgroundColor?: string;
    };
  };
};
```

#### Example Embed
```html
<meta property="fc:miniapp" content='{"version":"next","imageUrl":"https://example.com/og.png","button":{"title":"Play Game","action":{"type":"launch_miniapp","name":"My Game","url":"https://example.com/game","splashImageUrl":"https://example.com/splash.png","splashBackgroundColor":"#ffffff"}}}' />
```

**Key distinction**: Manifest = App's passport (one per domain). Embed = Content's business card (one per shareable page).

### `sdk.actions.ready()`
Hides the splash screen. **Must be called** or users see an infinite loading screen.

```ts
import { sdk } from '@farcaster/miniapp-sdk'
await sdk.actions.ready()
```

#### With splash config override:
```ts
await sdk.actions.ready({
  disableNativeGestures: true  // prevents swipe-to-close on mobile
})
```

### `sdk.actions.addMiniApp()`
Prompt the user to add the Mini App to their Farcaster client.


### `sdk.actions.signManifest()`
Sign the app manifest for domain verification.

```ts
import { sdk } from '@farcaster/miniapp-sdk'
import * as SignManifest from '@farcaster/miniapp-sdk'

try {
  const result = await sdk.actions.signManifest({ domain: 'example.com' })
  // result.signature, result.header, result.payload
} catch (error) {
  if (error instanceof SignManifest.RejectedByUser) {
    console.log('User rejected manifest signing')
  } else if (error instanceof SignManifest.InvalidDomain) {
    console.log('Invalid domain format')
  } else if (error instanceof SignManifest.GenericError) {
    console.log('Signing failed:', error.message)
  }
}
```

## Authentication — Quick Auth

Quick Auth is a lightweight service built on top of Sign In with Farcaster (SIWF) that provides easy authenticated sessions.

### Frontend — Make Authenticated Requests

```tsx
import React, { useState, useEffect } from "react";
import { sdk } from "@farcaster/miniapp-sdk";

export function App() {
  const [user, setUser] = useState<{ fid: number }>();

  useEffect(() => {
    (async () => {
      // sdk.quickAuth.fetch automatically adds Bearer token header
      const res = await sdk.quickAuth.fetch(`${BACKEND_ORIGIN}/me`);
      if (res.ok) {
        setUser(await res.json());
        sdk.actions.ready()
      }
    })()
  }, [])

  if (!user) return null;
  return <div>hello, {user.fid}</div>
}
```

### Frontend — Get Token Directly

```html
<script type="module">
  import { sdk } from "https://esm.sh/@farcaster/miniapp-sdk";
  const { token } = await sdk.quickAuth.getToken();
  // Use token as: Authorization: Bearer ${token}
</script>
```

### Backend — Validate Token

```bash
npm install @farcaster/quick-auth
```

```ts
import { Errors, createClient } from '@farcaster/quick-auth'

const client = createClient()

// In your middleware:
const payload = await client.verifyJwt({
  token: bearerToken,
  domain: 'your-app-domain.com',
})
// payload.sub = FID of authenticated user
```

### JWT Payload Structure
```json
{
  "iat": 1747764819,
  "iss": "https://auth.farcaster.xyz",
  "exp": 1747768419,
  "sub": 6841,               // FID of the user
  "aud": "miniapps.farcaster.xyz"
}
```

### Performance Optimization
```html
<link rel="preconnect" href="https://auth.farcaster.xyz" />
```

### SDK Quick Auth Methods
| Method | Description |
|--------|-------------|
| `sdk.quickAuth.fetch(url, options?)` | Make authenticated fetch (auto-adds Bearer token) |
| `sdk.quickAuth.getToken()` | Get/refresh a Quick Auth JWT |
| `sdk.quickAuth.token` | Returns current token if in memory (sync) |

## Notifications

### Overview
1. User enables notifications → Farcaster client generates a unique token → sends to your `webhookUrl`
2. To send notification → POST to the notification URL with the token
3. User disables notifications → your server receives an event → token is invalidated

### Setup: Add webhookUrl to Manifest
```json
{
  "miniapp": {
    "webhookUrl": "https://example.com/api/webhook"
  }
}
```

### Webhook Events

Your server receives POST requests to `webhookUrl` for these events:

#### `miniapp_added`
```json
{
  "event": "miniapp_added",
  "notificationDetails": {
    "url": "https://api.farcaster.xyz/v1/frame-notifications",
    "token": "a05059ef2415c67b08ecceb539201cbc6"
  }
}
```

#### `miniapp_removed`
```json
{ "event": "miniapp_removed" }
```

#### `notifications_enabled`
```json
{
  "event": "notifications_enabled",
  "notificationDetails": {
    "url": "https://api.farcaster.xyz/v1/frame-notifications",
    "token": "a05059ef2415c67b08ecceb539201cbc6"
  }
}
```

#### `notifications_disabled`
```json
{ "event": "notifications_disabled" }
```

### Verifying Webhook Events
```bash
npm install @farcaster/miniapp-node
```

```ts
import { parseWebhookEvent, verifyAppKeyWithNeynar } from "@farcaster/miniapp-node";

try {
  const data = await parseWebhookEvent(requestJson, verifyAppKeyWithNeynar);
  // data.event = event type
  // data.fid = user FID
} catch (e) {
  // Handle InvalidDataError, InvalidAppKeyError, etc.
}
```

### Sending Notifications

POST to the `url` from the notification token with:
```json
{
  "notificationId": "unique-id-for-deduplication",
  "title": "Notification Title",
  "body": "Notification body text",
  "targetUrl": "https://example.com/specific-page",
  "tokens": ["token1", "token2"]  // batch up to 100 tokens
}
```

> ⚠️ `targetUrl` hostname must EXACTLY match your registered domain. `www.example.com` ≠ `example.com`

#### Response Format
```json
{
  "result": {
    "successfulTokens": ["token1"],
    "invalidTokens": ["token2"],
    "rateLimitedTokens": []
  }
}
```

#### Rate Limits (Warpcast)
- 1 notification per 30 seconds per token
- 100 notifications per day per token
- Batch up to 100 tokens per request

#### Deduplication
Use stable `notificationId` values. The key `(fid, notificationId)` deduplicates over 24 hours. Example: `"daily-reminder-05-06-2024"`.

## Wallet Connections

### Ethereum Wallet

```ts
import { sdk } from '@farcaster/miniapp-sdk'

// Get an EIP-1193 compatible provider
const provider = await sdk.wallet.getEthereumProvider()

// Use with ethers.js
import { BrowserProvider } from 'ethers'
const ethersProvider = new BrowserProvider(provider)
const signer = await ethersProvider.getSigner()

// Use with viem
import { createWalletClient, custom } from 'viem'
import { base } from 'viem/chains'
const walletClient = createWalletClient({
  chain: base,
  transport: custom(provider)
})

// Request accounts
const accounts = await provider.request({ method: 'eth_requestAccounts' })
```

#### Supported RPC Methods
- `eth_requestAccounts`
- `eth_sendTransaction`
- `eth_signTypedData_v4`
- `personal_sign`
- `wallet_switchEthereumChain`
- Standard read methods via provider

### Solana Wallet

```ts
import { sdk } from '@farcaster/miniapp-sdk'

const provider = await sdk.wallet.getSolanaProvider()

// Connect
const { publicKey } = await provider.connect()

// Sign and send transaction
import { Transaction } from '@solana/web3.js'
const tx = new Transaction()
// ... build transaction ...
const { signature } = await provider.signAndSendTransaction(tx)

// Sign message
const message = new TextEncoder().encode('Hello Farcaster')
const { signature } = await provider.signMessage(message)

// Sign transaction (without sending)
const signedTx = await provider.signTransaction(tx)
```

#### Detecting Wallet Capabilities

```ts
import { sdk } from '@farcaster/miniapp-sdk'

// Check supported chains
const chains = await sdk.getChains()
// Returns CAIP-2 identifiers e.g. ["eip155:1", "eip155:8453", "solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp"]

// Check supported SDK methods
const capabilities = await sdk.getCapabilities()
// e.g. ["wallet.getEthereumProvider", "wallet.getSolanaProvider", "actions.composeCast", ...]
```

```ts
  console.log('User added the app')
})

sdk.on('miniappRemoved', () => {
  console.log('User removed the app')
})

sdk.on('notificationsEnabled', (data) => {
  // data.notificationDetails.token, data.notificationDetails.url
  console.log('Notifications enabled', data)
})

sdk.on('notificationsDisabled', () => {
  console.log('Notifications disabled')
})

// Cleanup
sdk.removeListener('miniappAdded', handler)
sdk.removeAllListeners()
```

## Publishing

### Steps
1. **Choose a stable domain** (can't be changed later; `www.example.com` ≠ `example.com`)
2. **Host manifest** at `/.well-known/farcaster.json`
3. **Create account association** using the [Manifest Tool](https://farcaster.xyz/~/developers/mini-apps/manifest)
4. **Register** with Farcaster
5. **Add embed meta tags** to pages you want shareable

### Account Association
The `accountAssociation` in your manifest cryptographically links your Farcaster FID to your domain. Generate it via the developer tools at https://farcaster.xyz/~/developers/mini-apps/manifest.

### Farcaster Hosted Manifests
Instead of self-hosting, redirect `/.well-known/farcaster.json` to Farcaster's hosted manifest service:

```ts
// Next.js
const nextConfig = {
  async redirects() {
    return [{
      source: '/.well-known/farcaster.json',
      destination: 'https://api.farcaster.xyz/miniapps/hosted-manifest/YOUR_ID',
      permanent: false,
    }]
  },
}
```

## Domain Migration (canonicalDomain)

To migrate your app to a new domain:

1. **New domain manifest** — full manifest with account association from same FID
2. **Old domain manifest** — add `canonicalDomain: "new-domain.com"`
3. **New domain manifest** — also add `canonicalDomain: "new-domain.com"` (self-referential)

```json
// Old domain: old-domain.com/.well-known/farcaster.json
{
  "miniapp": {
    "canonicalDomain": "new-domain.com"
  }
}

// New domain: new-domain.com/.well-known/farcaster.json
{
  "miniapp": {
    "canonicalDomain": "new-domain.com"
  }
}
```

**Rules:**
- ✅ `app.new-domain.com`
- ✅ `new-domain.com`
- ❌ `https://new-domain.com` (no protocol)
- ❌ `new-domain.com:3000` (no port)
- ❌ `new-domain.com/app` (no path)

> ⚠️ Keep the same `webhookUrl` in your new manifest to avoid breaking existing notification tokens.

## App Discovery & Search

### Requirements for Search Indexing
1. `noindex: true` NOT set in manifest
2. Manifest registered with Farcaster
3. Recent user activity and engagement
4. Working images (`iconUrl` must return valid `image/*` content-type)
5. Production domain (not ngrok, replit.dev, localtunnel, etc.)
6. Required fields: `name`, `iconUrl`, `homeUrl`, `description`

### Ranking Signals
- Number of users who opened the app
- Number of users who added the app to their collection
- EVM and SOL transaction volume within the app
- Trending score based on recent engagement

## Manifest vs Embed — Key Differences

| | Manifest | Embed |
|---|---|---|
| Location | `/.well-known/farcaster.json` | `<meta property="fc:miniapp">` on each page |
| Purpose | App identity & registration | Page-level social sharing card |
| Frequency | One per domain | One per shareable page |
| Required for | Notifications, app stores, adding app | Rich cards in social feeds |

## Key Notes & Gotchas

1. **MUST call `sdk.actions.ready()`** or users see infinite loading screen
2. **`www.example.com` ≠ `example.com`** — treated as completely separate apps
3. **`addMiniApp()` only works on production domain** — not tunnel URLs
4. **`targetUrl` in notifications must exactly match registered domain** — mismatches permanently invalidate tokens
5. **Frames v2 = Mini Apps** — same thing, rebranded
6. **Node.js 22.11.0+** required for SDK
7. **Quick Auth server**: `https://auth.farcaster.xyz`
8. **Webhook verification package**: `@farcaster/miniapp-node`
9. **Backend JWT validation package**: `@farcaster/quick-auth`
10. **Production domain required** for search indexing (no dev tunnels)
